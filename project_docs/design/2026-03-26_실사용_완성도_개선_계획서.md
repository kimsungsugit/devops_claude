# 설계 문서 - 문서 생성 및 영향도 분석 실사용 완성도 개선 계획서

## 1. 배경

- 현재 저장소에는 소스 변경 감지, 영향 함수 식별, UDS/SUTS/SITS 관련 처리, 영향도 감사 로그, 변경 이력 로그, 프론트엔드 결과 표시까지 주요 흐름이 이미 연결되어 있다.
- 최근 결과물을 보면 `reports/impact_audit/`, `reports/impact_changes/` 아래에 실제 실행 이력이 남아 있으며, 프론트엔드에서도 영향 요약과 리뷰 가이드가 노출되고 있다.
- 다만 현재 상태는 "구조가 연결된 상태"에 가깝고, "운영 가능한 수준으로 검증된 상태"라고 보기에는 아직 보완이 필요하다.
- 특히 최신 실행 이력 기준으로 `dry_run: true`가 포함되어 있고, 일부 문서 결과는 `review_required` 또는 `flagged` 중심으로 남아 있어 실제 산출물 갱신이 충분히 입증되지는 않았다.

## 2. 문제 정의

현재 실사용 완성도를 떨어뜨리는 핵심 문제는 아래 세 가지다.

- 첫째, 실제 실행 기준 검증 부족
  - `dry_run`이 아닌 실제 실행에서 UDS/SUTS/SITS 산출물이 기대대로 갱신되는지 확인이 부족하다.
  - change log에는 영향 함수 수가 기록되더라도, 실제 생성 산출물의 수치가 0이거나 review-only 상태로 남는 경우가 있다.

- 둘째, 대표 시나리오 기반의 end-to-end 검증 부족
  - 기능은 붙어 있으나, 운영자가 신뢰할 수 있는 대표 변경 유형별 재현 테스트 세트가 없다.
  - BODY 변경, HEADER 변경, 다중 파일 변경, 문서 재생성 제외 케이스 등이 체계적으로 검증되지 않았다.

- 셋째, 운영 인수인계 문서 부족
  - 현재 코드와 기능은 존재하지만, 신규 운영자 또는 다른 개발자가 그대로 이어받아 실행하기 위한 README와 사용자 가이드가 정돈되어 있지 않다.
  - 특히 루트 `README.md`는 인코딩과 설명 품질 문제로 저장소 진입 문서 역할을 충분히 하지 못한다.

## 3. 계획 목적

본 계획의 목적은 다음과 같다.

- `dry_run`이 아닌 실제 실행 기준으로 문서 생성과 영향도 분석 파이프라인이 정상 동작함을 입증한다.
- 대표 변경 시나리오 3~5개를 기반으로 end-to-end 재현 테스트 체계를 확보한다.
- README와 사용자 가이드를 정리하여 운영 인수인계가 가능한 상태로 만든다.
- 최종적으로 "기능 구현 완료"가 아니라 "운영 가능한 수준의 실사용 완성도 확보"를 달성한다.

## 4. 범위

본 계획서의 범위는 다음과 같다.

- 영향도 분석 실행 경로
  - 로컬 영향도 트리거
  - 영향 함수 식별
  - audit/change log 저장
  - 프론트엔드 결과 표시

- 문서 생성/갱신 대상
  - UDS
  - SUTS
  - SITS

- 검증 및 운영 문서
  - 루트 `README.md`
  - 실무 사용자용 가이드
  - 대표 시나리오 테스트 체크리스트

범위 밖 항목은 다음과 같다.

- 대규모 UI 재설계
- 전체 아키텍처 재작성
- SDS/STS 자동 생성 정책의 전면 개편
- 문서 템플릿 자체의 대규모 개편

## 5. 현재 상태 평가

### 5.1 구현 완료도가 높은 영역

- 백엔드에 영향도 분석 오케스트레이션 흐름이 연결되어 있다.
- 영향 분석 결과가 `impact_audit`, `impact_changes` 형태로 파일로 남는다.
- 프론트엔드에서 영향 요약, 리뷰 대상, 변경 로그 접근이 가능하다.
- SITS에 대해서는 VectorCAST export 패키지 경로까지 마련되어 있다.

### 5.2 부분 완료 상태인 영역

- 영향 함수 집계는 되고 있으나, 실제 산출물 재생성 결과가 최신 로그에서 충분히 강하게 나타나지 않는다.
- 최신 샘플에서는 `dry_run: true` 상태가 확인되어 운영 기준 검증으로 보기 어렵다.
- 일부 문서 항목은 `review_required` 위주로 남아 있어 자동 갱신 범위와 정책이 명확히 닫히지 않았다.

### 5.3 미완료 영역

- 대표 시나리오 기반 acceptance test 세트 부재
- 운영자 관점 실행 가이드 부재
- README 정리 부족
- "언제 regenerate, 언제 review-only"인지에 대한 운영 기준 문서화 부족

## 6. 개선 목표

### 목표 1. 실제 실행 기준 산출물 갱신 검증

다음 상태를 달성해야 한다.

- `dry_run = false` 기준 실행 결과를 확보한다.
- UDS/SUTS/SITS 중 자동 생성 대상은 실제 산출물이 갱신되어야 한다.
- change log 요약 수치가 모두 0으로만 남지 않아야 하며, 영향도와 산출물 갱신의 관계가 설명 가능해야 한다.
- review-only로 남는 경우에는 "왜 자동 갱신하지 않았는지"가 분명해야 한다.

### 목표 2. 대표 시나리오 기반 end-to-end 재현 테스트 확보

다음 상태를 달성해야 한다.

- 최소 3개, 가능하면 5개 시나리오를 정의한다.
- 각 시나리오마다 입력 변경, 실행 방법, 기대 결과, 검증 포인트, 실패 판정 기준이 있어야 한다.
- 누구나 같은 시나리오를 반복 실행해 동일한 검증을 수행할 수 있어야 한다.

### 목표 3. 운영 인수인계 가능한 README 및 사용자 가이드 정리

다음 상태를 달성해야 한다.

- 신규 담당자가 저장소를 열고 README만 봐도 전체 구조와 실행 흐름을 이해할 수 있어야 한다.
- 실제 운영자가 문서 생성/영향도 분석을 어떻게 실행하고 어디서 결과를 확인하는지 알 수 있어야 한다.
- 예외 상황과 점검 포인트가 사용자 가이드에 포함되어야 한다.

## 7. 상세 개선 계획

## 7.1 실제 실행 기준 산출물 갱신 검증 계획

### 7.1.1 목적

- 현재 구조가 단지 영향 함수만 계산하는 수준이 아니라, 실제로 문서 산출물 갱신까지 연결되는지 검증한다.

### 7.1.2 핵심 개선 항목

- `dry_run = false` 실행 경로를 우선 검증한다.
- `auto_generate = true` 조건에서 UDS/SUTS/SITS가 어떤 규칙으로 생성되는지 확인한다.
- 생성 후 결과 파일 경로와 linked docs 갱신 여부를 확인한다.
- change log의 숫자가 실제 파일 생성 결과와 일치하는지 점검한다.

### 7.1.3 수행 항목

1. 실행 전 기준선 확보
   - 현재 linked docs 경로를 기록한다.
   - 기존 UDS/SUTS/SITS 최신 파일의 생성 시각과 경로를 기록한다.
   - 영향 분석 전 상태 스냅샷을 남긴다.

2. 실제 실행 수행
   - `dry_run = false`
   - `auto_generate = true`
   - 대표 SCM ID와 source root를 기준으로 실행한다.

3. 결과 확인
   - 새 audit 파일 생성 여부 확인
   - 새 change log 파일 생성 여부 확인
   - UDS/SUTS/SITS 산출물 신규 생성 또는 갱신 여부 확인
   - linked docs 갱신 여부 확인
   - 프론트엔드와 change log 수치 일치 여부 확인

4. 예외 분류
   - 산출물이 생성되지 않은 경우
   - 생성은 되었지만 수치가 0인 경우
   - review_required로 떨어진 경우
   - 산출물은 생성되었으나 내용이 영향 함수와 맞지 않는 경우

### 7.1.4 완료 기준

- 최소 1회 이상 실제 실행에서 UDS/SUTS/SITS 중 대상 산출물이 갱신된다.
- change log summary에 의미 있는 비영(0이 아닌) 수치가 기록된다.
- review-only 결과는 정책적으로 설명 가능하다.
- 결과 파일 경로와 로그 값이 서로 모순되지 않는다.

## 7.2 대표 변경 시나리오 기반 end-to-end 재현 테스트 계획

### 7.2.1 목적

- 운영자가 "어떤 변경에 대해 이 시스템이 어디까지 반응하는지"를 신뢰할 수 있도록 대표 시나리오 세트를 만든다.

### 7.2.2 권장 시나리오 구성

아래 5개를 권장한다.

1. 시나리오 A: BODY 변경 단일 함수 케이스
   - 목적: 내부 로직 수정 시 영향 함수 집계와 문서/테스트 갱신 반응 검증
   - 기대 포인트: BODY 분류, 직접 영향 함수 식별, SUTS/SITS 반영 여부 확인

2. 시나리오 B: HEADER 또는 SIGNATURE 변경 케이스
   - 목적: 인터페이스 변경 시 UDS 및 연계 테스트 문서 영향 범위 확대 여부 검증
   - 기대 포인트: UDS 우선 반응, STS/SDS review flag 적절성 확인

3. 시나리오 C: 다중 파일 동시 변경 케이스
   - 목적: 모듈 단위 변경 시 change log 집계, 영향 함수 수 확장, 로그 가독성 검증
   - 기대 포인트: direct/indirect impact 분리, 결과 로그 누락 없음

4. 시나리오 D: 변경은 있으나 재생성 제외가 맞는 케이스
   - 목적: 모든 변경이 자동 생성으로 이어지지 않아야 하는 정책 검증
   - 기대 포인트: review_required 또는 flag 처리의 정당성 확인

5. 시나리오 E: SITS 중심 통합 영향 케이스
   - 목적: 호출 흐름 영향이 있는 경우 SITS 및 VectorCAST export 연계 검증
   - 기대 포인트: integration flow 반영, export package 생성 가능 여부 확인

### 7.2.3 시나리오별 공통 템플릿

각 시나리오는 아래 형식으로 정리한다.

- 시나리오명
- 변경 파일
- 변경 함수
- 변경 유형 예상값
- 실행 조건
- 실행 명령 또는 UI 경로
- 기대 산출물
- 기대 change log 값
- 기대 review 문서 생성 여부
- 실패 조건
- 비고

### 7.2.4 수행 방식

- 시나리오 문서는 `project_docs/design` 또는 별도 `project_docs/change_requests` 아래 관리한다.
- 최초 1회는 수동 검증 중심으로 수행한다.
- 검증이 안정화되면 반복 가능한 스모크 테스트 세트로 정리한다.
- 각 시나리오 결과는 pass/fail과 증적 파일 경로를 함께 기록한다.

### 7.2.5 완료 기준

- 최소 3개 시나리오, 목표 5개 시나리오에 대해 결과가 기록된다.
- 각 시나리오별 audit/change log/산출물 경로가 추적 가능하다.
- 동일 절차로 재실행했을 때 운영자가 동일한 검증을 수행할 수 있다.

## 7.3 README 및 사용자 가이드 정리 계획

### 7.3.1 목적

- 코드 작성자가 아닌 운영자, 신규 개발자, 인수인계 대상자가 시스템을 이해하고 실행할 수 있도록 문서를 정리한다.

### 7.3.2 README 개선 방향

루트 `README.md`는 아래 구조로 재정리한다.

- 프로젝트 개요
- 현재 지원 기능
  - UDS 생성
  - STS/SUTS/SITS 생성
  - 영향도 분석
  - Jenkins/로컬 흐름
- 시스템 구성
  - frontend
  - backend
  - workflow
  - generators
  - reports
- 빠른 시작
  - backend 실행
  - frontend 실행
  - 주요 화면 진입
- 실사용 핵심 흐름
  - 로컬 영향도 실행
  - 결과 확인 위치
  - 생성 산출물 위치
- 주요 디렉터리 설명
- 운영 시 주의사항
- 문제 발생 시 확인 포인트

### 7.3.3 사용자 가이드 개선 방향

별도 사용자 가이드는 운영 관점으로 작성한다.

- 언제 사용하는가
  - 소스 변경 후 영향 분석 필요 시
  - 문서 갱신 필요 시
  - 테스트 문서 영향 검토 시

- 어떻게 실행하는가
  - 필수 입력값
  - UI에서 실행하는 방법
  - 필요한 경우 API 또는 내부 실행 포인트

- 결과는 어디서 보는가
  - audit 위치
  - change log 위치
  - UDS/SUTS/SITS 산출물 위치
  - 프론트엔드 확인 위치

- 결과는 어떻게 해석하는가
  - direct / indirect impact 의미
  - review_required 의미
  - flagged 의미
  - generated artifact 의미

- 실패 시 어떻게 대응하는가
  - 수치가 0인 경우
  - 산출물이 생성되지 않는 경우
  - linked doc가 갱신되지 않는 경우
  - preview/viewer가 비어 있는 경우

### 7.3.4 완료 기준

- README 인코딩 문제가 제거된다.
- README만으로 시스템 구조와 실행 진입점을 이해할 수 있다.
- 사용자 가이드만 보고도 운영자가 1회 실행과 결과 확인을 수행할 수 있다.

## 8. 우선순위

우선순위는 아래와 같다.

1. 실제 실행 검증
   - 가장 먼저 해결해야 한다.
   - 산출물 갱신이 증명되지 않으면 나머지 문서화는 신뢰를 주기 어렵다.

2. 대표 시나리오 확보
   - 실제 실행이 가능해진 뒤 바로 시나리오화해야 한다.
   - 그래야 기능이 다시 깨졌을 때 빠르게 확인할 수 있다.

3. README 및 사용자 가이드 정리
   - 위 두 항목의 결과를 반영해 최종 정리해야 한다.
   - 운영 문서는 실제 동작이 확인된 흐름을 기준으로 써야 한다.

## 9. 단계별 실행 순서

### 1단계. 실제 실행 검증

- linked docs 및 기존 산출물 기준선 정리
- non-dry-run 실행
- 결과 산출물 및 change log 확인
- 문제 유형 분류

### 2단계. 시나리오 세트 구축

- 대표 시나리오 3~5개 선정
- 각 시나리오의 입력/기대값/판정 기준 정의
- 시나리오별 실행 및 증적 기록

### 3단계. 운영 문서 정리

- README 정비
- 사용자 가이드 작성
- 결과 해석 및 장애 대응 절차 정리

### 4단계. 최종 acceptance 점검

- 실제 실행 증적 확인
- 시나리오 pass/fail 정리
- 인수인계 가능 여부 최종 판단

## 10. 산출물 계획

본 개선 작업의 산출물은 다음과 같다.

- 실제 실행 검증 결과 기록서
- 대표 시나리오 테스트 목록 및 결과표
- 정비된 `README.md`
- 운영자용 사용자 가이드
- 최종 acceptance 체크리스트

## 11. 수용 기준

아래 조건을 만족하면 본 개선 작업을 완료로 본다.

- `dry_run = false` 기준 실행 증적이 있다.
- UDS/SUTS/SITS 대상 중 기대되는 산출물 갱신이 실제로 확인된다.
- change log summary가 실제 결과를 반영하는 비영 값으로 기록된다.
- 최소 3개 이상의 대표 시나리오가 pass/fail과 함께 문서화된다.
- README와 사용자 가이드만으로 기본 운영 흐름을 재현할 수 있다.

## 12. 리스크 및 대응

- 리스크: 영향 함수는 잡히지만 산출물 생성으로 이어지지 않을 수 있다.
  - 대응: auto-generate 조건과 linked docs 갱신 규칙을 먼저 점검한다.

- 리스크: review_required가 과도하게 사용되어 실제 생성 부족을 가릴 수 있다.
  - 대응: review-only 허용 조건을 문서로 고정한다.

- 리스크: 시나리오가 특정 프로젝트 상태에만 맞고 재현성이 떨어질 수 있다.
  - 대응: 입력 파일, 기준 브랜치, 실행 조건을 명시한다.

- 리스크: README가 기능 목록만 나열하고 운영 설명이 부족할 수 있다.
  - 대응: "어떻게 실행하고 어디서 결과를 보는가" 중심으로 재작성한다.
