# 설계 문서 - 대표 변경 시나리오 기반 End-to-End 재현 테스트 계획서

## 1. 배경

- 현재 시스템은 소스 변경 감지, 영향 함수 식별, 문서 생성/리뷰 대상 판정, audit/change log 저장, 프론트엔드 결과 표시까지 주요 흐름이 연결되어 있다.
- 그러나 실사용 완성도를 높이기 위해서는 "기능이 있다"는 수준을 넘어, 대표적인 변경 유형에 대해 반복 가능한 방식으로 같은 결과를 재현할 수 있어야 한다.
- 특히 운영자나 인수인계 대상자는 소스 변경이 들어왔을 때 어떤 유형의 반응이 정상인지 판단할 기준이 필요하다.
- 이를 위해 대표 변경 시나리오 3~5개를 정의하고, 각 시나리오에 대해 end-to-end 재현 테스트 절차를 마련해야 한다.

## 2. 목적

본 계획서의 목적은 다음과 같다.

- 대표 변경 유형별로 시스템의 기대 반응을 명확히 정의한다.
- 소스 변경부터 impact 결과, change log, 문서 산출물, 프론트엔드 표시까지 전체 흐름을 검증하는 재현 테스트 체계를 만든다.
- 운영자, 개발자, 인수인계 대상자가 동일한 절차로 같은 검증을 수행할 수 있도록 한다.
- 기능 회귀가 발생했을 때 빠르게 이상 여부를 판단할 수 있는 기준선을 만든다.

## 3. 범위

본 계획서의 범위는 다음과 같다.

- 입력
  - 대표적인 소스 변경 유형
  - 변경 파일 목록
  - 변경 함수 유형

- 처리 흐름
  - 영향도 분석 실행
  - direct/indirect impact 계산
  - change log 생성
  - audit 파일 생성
  - UDS/SUTS/SITS 산출물 생성 또는 review-required 판정
  - 프론트엔드 결과 확인

- 출력
  - 시나리오별 pass/fail 결과
  - 증적 파일 경로
  - 기대 결과 대비 실제 결과 비교표

범위 밖 항목은 다음과 같다.

- 전체 UI 재설계
- 대규모 코드 리팩터링
- 문서 생성 엔진 자체의 구조 개편
- Jenkins 중심 복합 배포 검증

## 4. 왜 대표 시나리오가 필요한가

현재와 같은 시스템은 단일 기능 테스트만으로 신뢰를 확보하기 어렵다. 이유는 다음과 같다.

- 영향 함수 수가 맞아도 실제 문서 산출물이 안 바뀔 수 있다.
- change log가 생성되어도 프론트엔드에 반영이 안 될 수 있다.
- review-required가 정상 fallback인지, 단순 실패인지 구분이 필요하다.
- 특정 변경 유형에서는 UDS만 반응해야 하고, 다른 유형에서는 SUTS/SITS까지 반응해야 할 수 있다.

즉, "대표 변경 유형별 정상 반응"을 정의하고 그 반응을 반복 확인할 수 있어야 운영 신뢰도가 올라간다.

## 5. 테스트 기본 원칙

대표 변경 시나리오 기반 end-to-end 테스트는 아래 원칙으로 설계한다.

- 실제 운영 흐름과 최대한 유사해야 한다.
- 입력 변경과 기대 결과가 명확해야 한다.
- 수동 확인이 필요한 부분과 자동 확인 가능한 부분을 구분해야 한다.
- 실패 시 어느 단계에서 깨졌는지 추적 가능해야 한다.
- 한 번만 되는 테스트가 아니라 반복 재현 가능한 테스트여야 한다.

## 6. 검증 대상 흐름

각 시나리오는 아래 흐름 전체를 검증한다.

1. 소스 변경 준비
2. 영향도 분석 실행
3. changed_files / changed_functions 결과 확인
4. audit 파일 생성 여부 확인
5. change log 생성 여부 확인
6. UDS/SUTS/SITS 생성 또는 review-required 판정 확인
7. linked docs 또는 출력 파일 경로 확인
8. 프론트엔드 영향도 화면 확인
9. 결과를 기준 기대값과 비교
10. pass/fail 판정 및 증적 기록

## 7. 대표 시나리오 구성안

최소 3개, 권장 5개 시나리오를 운영 기준으로 제안한다.

## 7.1 시나리오 A - BODY 변경 단일 함수 케이스

### 목적

- 함수 내부 로직만 바뀌는 가장 일반적인 변경에서 시스템이 정상적으로 반응하는지 확인한다.

### 대상 변경

- `.c` 파일 내 특정 함수 BODY만 수정
- 함수 시그니처, 헤더, 외부 인터페이스는 변경하지 않음

### 기대 결과

- changed function type이 `BODY`로 식별되어야 한다.
- direct impact 함수가 최소 1개 이상 잡혀야 한다.
- audit, change log가 생성되어야 한다.
- UDS는 review-required 또는 재생성 중 정책에 맞는 결과가 나와야 한다.
- SUTS/SITS는 BODY 변경 정책에 따라 영향 반영 또는 review flag가 보여야 한다.
- 프론트엔드에 BODY 변경 기반 가이드가 노출되어야 한다.

### 검증 포인트

- BODY 분류 정확성
- 영향 함수 누락 여부
- change log summary의 0이 아닌 값 존재 여부
- 프론트엔드 안내 문구와 실제 결과의 일치 여부

## 7.2 시나리오 B - HEADER 또는 SIGNATURE 변경 케이스

### 목적

- 인터페이스 변경 시 영향 범위가 BODY 변경보다 넓게 반응하는지 확인한다.

### 대상 변경

- 함수 선언부 변경
- 파라미터 추가/삭제/타입 변경
- 헤더 파일 선언 변경

### 기대 결과

- changed function type이 `HEADER` 또는 `SIGNATURE`로 식별되어야 한다.
- 관련 함수 및 연계 문서 영향 범위가 BODY 변경보다 커야 한다.
- UDS는 우선 반응 대상으로 표시되어야 한다.
- STS/SDS가 review 대상이라면 그 이유가 일관되게 드러나야 한다.
- 프론트엔드에 인터페이스 변경 중심 가이드가 노출되어야 한다.

### 검증 포인트

- 인터페이스 변경 타입 분류 정확성
- 영향 함수 수 증가 여부
- review guidance의 적절성
- change log의 문서별 flagged 수치 일관성

## 7.3 시나리오 C - 다중 파일 동시 변경 케이스

### 목적

- 실제 운영에서 자주 발생하는 다중 파일 변경 상황에서 집계와 로그 품질을 검증한다.

### 대상 변경

- 동일 모듈 또는 연관 모듈의 `.c` 파일 2~3개 이상 동시 변경
- 함수 여러 개가 BODY 또는 혼합 타입으로 변경

### 기대 결과

- changed_files 목록이 모두 반영되어야 한다.
- changed_functions 집계가 누락 없이 기록되어야 한다.
- direct/indirect impact 카운트가 분리되어 보여야 한다.
- change log summary가 실제 변경 규모를 반영해야 한다.
- 프론트엔드에서 로그와 동일한 수치를 보여야 한다.

### 검증 포인트

- 다중 파일 집계 누락 여부
- direct와 indirect count 신뢰성
- 로그 가독성
- 대량 영향 함수에서도 UI가 정상 표시되는지 여부

## 7.4 시나리오 D - Review-Only가 정상인 케이스

### 목적

- 모든 변경이 자동 재생성으로 이어지면 안 되므로, review-required fallback이 정상 동작하는지 검증한다.

### 대상 변경

- 자동 생성 정책상 문서 재생성보다 검토가 우선인 변경
- 또는 산출물 생성 없이 영향도 분석과 리뷰 가이드만 남는 것이 맞는 케이스

### 기대 결과

- 결과가 단순 실패가 아니라 정책상 review-required로 떨어져야 한다.
- audit/change log는 정상 생성되어야 한다.
- review 문서가 생성된다면 해당 문서에 변경 함수와 리뷰 포인트가 기록되어야 한다.
- 운영자가 "왜 재생성되지 않았는지" 설명 가능한 상태여야 한다.

### 검증 포인트

- review-required와 실패의 구분 가능성
- 리뷰 문서 내용 품질
- 운영 판단 근거 제공 여부

## 7.5 시나리오 E - SITS 중심 통합 영향 케이스

### 목적

- 호출 흐름 변화가 통합 테스트 관점에 반영되는지 확인한다.

### 대상 변경

- 호출 체인에 영향을 주는 함수 변경
- integration flow에 연쇄 영향을 주는 변경

### 기대 결과

- 영향 함수 외에 통합 경로 관련 정보가 SITS 측면에서 반영되어야 한다.
- SITS 생성 또는 review 결과가 남아야 한다.
- 가능하면 VectorCAST export까지 후속 검증 가능해야 한다.
- call chain 또는 integration context가 결과 해석에 활용 가능해야 한다.

### 검증 포인트

- SITS flagged 함수 반영 여부
- SITS 산출물 생성 또는 review 결과 적절성
- export 후속 가능성

## 8. 시나리오별 표준 템플릿

각 시나리오는 아래 템플릿으로 문서화한다.

- 시나리오 ID
- 시나리오명
- 목적
- 사전 조건
- 기준 브랜치 또는 기준 상태
- 변경 파일
- 변경 함수
- 예상 변경 유형
- 실행 방법
- 기대 결과
- 확인해야 할 파일
- 프론트엔드 확인 포인트
- 실패 조건
- 증적 파일 경로
- 최종 판정

## 9. 테스트 절차 상세

## 9.1 사전 준비

- 테스트 대상 저장소 상태를 고정한다.
- 기준 branch 또는 기준 commit을 정한다.
- linked docs 현재 상태를 기록한다.
- 기존 산출물 최신 경로를 기록한다.
- 테스트 실행 전 `reports/impact_audit`, `reports/impact_changes` 최신 파일 목록을 메모한다.

## 9.2 변경 적용

- 시나리오 정의에 맞는 변경만 최소 단위로 적용한다.
- 변경 범위를 불필요하게 넓히지 않는다.
- 변경 내용은 추후 재현 가능하도록 diff 또는 설명으로 저장한다.

## 9.3 영향도 분석 실행

- 로컬 UI 또는 API 기준으로 동일 절차를 사용한다.
- 가능하면 `dry_run = false` 기준 실행을 우선한다.
- `auto_generate` 여부를 시나리오별로 명시한다.

## 9.4 결과 수집

아래 결과를 모두 수집한다.

- audit 파일 경로
- change log 경로
- changed_files
- changed_functions
- impact_counts
- documents summary
- 생성된 UDS/SUTS/SITS 경로
- review-required markdown 경로
- 프론트엔드 캡처 또는 확인 결과

## 9.5 판정

각 시나리오는 아래 관점으로 판정한다.

- 입력 변경이 기대 타입으로 분류되었는가
- 영향 함수가 정책상 타당하게 계산되었는가
- audit/change log가 정상 생성되었는가
- 문서 생성 또는 review 결과가 정책과 일치하는가
- 프론트엔드 표시가 백엔드 로그와 일치하는가

최종 판정은 `PASS`, `PARTIAL PASS`, `FAIL` 세 단계로 한다.

## 10. PASS / FAIL 기준

## 10.1 PASS 기준

- 시나리오 입력이 재현 가능하다.
- 기대한 변경 유형 분류가 맞다.
- audit와 change log가 생성된다.
- 문서 결과 또는 review 결과가 정책에 맞는다.
- 프론트엔드 표시와 파일 로그가 일치한다.
- 운영자가 결과를 설명할 수 있다.

## 10.2 PARTIAL PASS 기준

- 영향 분석과 로그 생성은 정상이나, 일부 산출물 생성 또는 UI 표시가 기대와 다르다.
- 결과 해석은 가능하나 후속 수정이 필요하다.

## 10.3 FAIL 기준

- 변경 유형이 잘못 분류된다.
- audit 또는 change log가 생성되지 않는다.
- 영향 함수가 비정상적으로 비어 있다.
- 문서 생성/리뷰 결과가 정책과 맞지 않는다.
- 프론트엔드와 백엔드 로그가 서로 다르다.

## 11. 증적 관리 계획

시나리오 테스트는 단순 실행으로 끝내지 않고 증적을 남겨야 한다.

- 각 시나리오마다 아래를 보관한다.
  - 실행 일시
  - 기준 상태
  - 적용 diff 요약
  - audit 파일 경로
  - change log 경로
  - 생성 산출물 경로
  - 프론트엔드 확인 결과
  - 최종 판정

- 증적 문서는 Markdown으로 관리한다.
- 필요하면 시나리오별 결과 문서를 `project_docs/design` 또는 `project_docs/change_requests` 하위에 저장한다.

## 12. 운영 적용 방식

이 테스트 세트는 1회성 검증이 아니라 다음 용도로 활용한다.

- 신규 기능 반영 후 회귀 점검
- 영향도 분석 로직 수정 후 비교 검증
- 운영 이슈 발생 시 기준 시나리오 재현
- 인수인계 시 시스템 신뢰도 설명 자료

## 13. 우선순위

시나리오 우선순위는 아래 순서로 권장한다.

1. 시나리오 A
   - 가장 흔한 변경이며 기본 동작 확인에 적합하다.

2. 시나리오 B
   - 인터페이스 변경은 영향도 품질을 판단하는 핵심 케이스다.

3. 시나리오 C
   - 운영 현실에 가까운 복합 변경 검증이다.

4. 시나리오 D
   - review-only 정책이 정상인지 확인해야 운영 혼선을 줄일 수 있다.

5. 시나리오 E
   - SITS 및 통합 테스트 연계 품질을 높이는 고도화 검증이다.

## 14. 단계별 실행 계획

### 1단계. 시나리오 정의

- 대표 시나리오 5개 초안 확정
- 각 시나리오의 입력 변경 방식과 기대 결과 정의
- 표준 템플릿 작성

### 2단계. 1차 수동 검증

- 시나리오 A, B, C 우선 수행
- 결과 차이를 분석하고 판정 기준 보정

### 3단계. 정책 확인

- review-required 허용 조건 정리
- regenerate와 review-only 경계를 명확화

### 4단계. 시나리오 확장

- 시나리오 D, E 수행
- SITS 및 통합 영향 해석 기준 보강

### 5단계. 운영 문서화

- 시나리오별 결과 요약표 작성
- 운영자가 재실행 가능한 절차서로 정리

## 15. 최종 산출물

본 계획이 완료되면 아래 산출물이 있어야 한다.

- 대표 시나리오 정의서
- 시나리오별 실행 결과서
- pass/fail 기준표
- 증적 경로 목록
- 운영용 재현 테스트 절차서

## 16. 수용 기준

아래 조건을 만족하면 본 계획의 목표를 달성한 것으로 본다.

- 최소 3개, 목표 5개 대표 시나리오가 문서화된다.
- 각 시나리오가 동일 절차로 재현 가능하다.
- audit/change log/산출물/프론트 결과를 묶어 검증할 수 있다.
- PASS / PARTIAL PASS / FAIL 기준이 명확하다.
- 운영자가 문서만 보고도 시나리오를 재실행할 수 있다.

## 17. 결론

- 대표 변경 시나리오 기반 end-to-end 재현 테스트는 현재 시스템의 실사용 신뢰도를 높이기 위한 핵심 작업이다.
- 이 작업의 목적은 단순 테스트 추가가 아니라, "변경 유형별 정상 반응을 기준화"하는 데 있다.
- 따라서 본 계획은 코드 수정보다도 운영 검증 체계와 기준선 확보에 중심을 두어야 하며, 이후 기능 개선의 기준점 역할을 하게 된다.
