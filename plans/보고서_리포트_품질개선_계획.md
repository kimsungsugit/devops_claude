### 1.2 주요 발견 사항

**A. 백엔드 (보고서 생성) 문제:**
1. **`analysis_summary.md`가 너무 간소** - 단 38줄, 각 단계별 상세 수치/결론 없음
2. **Clang-Tidy 결과 파일 미생성** - 메모리에서만 처리, JSON 파일 없음
3. **Fuzzing 통합 요약 리포트 없음** - 7개 개별 result.json만 존재
4. **Domain Tests 에러 5건** - "no scenarios produced by agents" → AI 모델 호출 실패
5. **QEMU 리포트 부실** - meta.json만 있고, 결과 해석/요약 없음
6. **coverage JSON의 line_rate가 72.8%** vs gcovr 텍스트 출력 94% → auto_generated 포함 여부 차이 (gcovr `--exclude` 차이)
7. **Stub Injection(Step 1) 추적 없음** - 어떤 스텁이 주입되었는지 기록 미비

**B. 프론트엔드 (UI 표시) 문제:**
1. **Domain Tests 결과 UI 미표시** - 설정 옵션은 있으나 표시 컴포넌트 없음
2. **Fuzzing 상세 뷰 부족** - iterations 수, mode(dumb_fuzzer/libfuzzer), 타겟별 duration 미표시
3. **QEMU soft-fail 설명 부족** - "WARN" 표시만, RP2040 호환성 문제 설명 없음
4. **커버리지 파일별 상세 테이블 불완전** - coverage.xml 기반 파일별 데이터 미파싱
5. **테스트 실행 상세(CTest 결과) 표시 부족** - r1/r2 라운드 구분, 각 테스트 개별 결과 미표시

**C. 보고서 품질 문제:**
1. **최종 DOCX/XLSX 리포트 연동 부재** - 로컬 워크플로우 결과 → 문서화 자동 연결 없음
2. **이력 관리** - 이전 실행과 비교 기능 없음 (delta 보고)
3. **품질 게이트 판정 근거 부족** - 왜 통과/실패인지 상세 설명 없음

## 2. 개선 계획

### Phase 1: 백엔드 - 보고서 생성 보완 (우선순위: 높음)

#### 1-1. `analysis_summary.md` 상세화
- **현재**: 38줄 간소한 요약
- **목표**: 각 Step별 섹션 포함, 수치/판정/세부사항 기록
- **작업**:
  - `report_generator.py`의 `generate_markdown_summary()` 함수 확장
  - Coverage: 파일별 line/branch/func 커버리지 테이블 추가
  - Fuzzing: 타겟별 iterations, mode, duration, crash 여부 테이블
  - Static Analysis: 도구별 이슈 수, 심각도별 분류 테이블
  - Build/Test: CTest 결과 테이블 (테스트명, 상태, 소요시간)
  - QEMU: 실행 결과, soft-fail 사유 설명
  - Domain Tests: 성공/에러 목록
  - 최종 품질 점수 (risk_score) 근거 설명

#### 1-2. Clang-Tidy 결과 파일 생성
- **현재**: 메모리 내 Issue 리스트만 존재
- **목표**: `reports/clang_tidy/clang_tidy_findings.json` 파일 생성
- **작업**:
  - `pipeline.py`의 Clang-Tidy 단계에서 결과를 JSON으로 직렬화
  - `findings_flat.json`과 동일한 포맷으로 개별 파일도 저장

#### 1-3. Fuzzing 통합 요약 리포트
- **현재**: 타겟별 `fuzz_*_result.json` 개별 파일만 존재
- **목표**: `reports/fuzz/fuzzing_summary.json` + `fuzzing_summary.md` 생성
- **작업**:
  - 전체 타겟 결과 집계 (총 iterations, 총 duration, crash 수)
  - 타겟별 요약 테이블 (MD)
  - focus vs non-focus 구분

#### 1-4. Coverage 수치 정확성 보정
- **현재**: `analysis_summary.json`의 `line_rate=72.8%` vs 실제 libs만 94%
- **목표**: libs 소스만의 커버리지와 전체 커버리지 모두 보고
- **작업**:
  - `pipeline.py`에서 gcovr `--exclude` 적용 후 libs 전용 rate 추가
  - `coverage_summary`에 `libs_line_rate`, `libs_branch_rate` 필드 추가
  - MD 리포트에 "Production Code Coverage" vs "Total Coverage" 구분 표시

#### 1-5. Domain Tests 에러 해결
- **현재**: 5개 타겟에서 "no scenarios produced by agents" 에러
- **목표**: AI 모델 호출 성공하여 도메인 테스트 시나리오 생성
- **작업**:
  - `--agent-run-mode off`에서도 domain test LLM 호출이 올바른지 확인
  - 기존 `tests/domain/test_e2e_domain.c` 등 이미 존재하는 테스트 감지 로직 확인
  - 에러 로그 상세 기록 (API 응답, 모델명, timeout 등)

#### 1-6. QEMU 리포트 강화
- **현재**: `qemu_smoke.meta.json`, stderr 로그만 존재
- **목표**: `reports/qemu/qemu_summary.json` + 상세 결과 보고
- **작업**:
  - ELF 아키텍처 정보, 사용된 QEMU 머신 기록
  - soft-fail 판정 사유 명시 (RP2040 ↔ lm3s6965evb 불일치)
  - 에러 패턴 분석 결과 포함 (Lockup, HardFault 등)

### Phase 2: 프론트엔드 - UI 보고서 표시 개선 (우선순위: 중간)

#### 2-1. Domain Tests 결과 UI 컴포넌트 추가
- **현재**: UI에 표시 코드 없음
- **목표**: Testing 탭에 Domain Tests 섹션 추가
- **작업**:
  - `LocalWorkflow.jsx` Testing 탭에 domain_tests 결과 패널
  - 성공 테스트: 파일명, 시나리오 수, 실행 결과
  - 에러 테스트: 타겟, 에러 원인 표시
  - 도메인 테스트 파일 에디터 열기 링크

#### 2-2. Fuzzing 상세 뷰 강화
- **현재**: 크래시 여부만 간략 표시
- **목표**: 타겟별 상세 테이블, iterations, mode 표시
- **작업**:
  - 타겟별 카드: iterations, duration, mode(dumb_fuzzer/libfuzzer)
  - focus 타겟 하이라이트
  - harness 소스 뷰 링크
  - 로그 뷰 (stdout/stderr)

#### 2-3. QEMU 결과 상세 표시
- **현재**: "WARN" 상태만 표시
- **목표**: soft-fail 사유, 레지스터 덤프, 에러 패턴 표시
- **작업**:
  - soft-fail 배지 + 툴팁 (RP2040 호환성 설명)
  - 레지스터 상태 접을 수 있는 상세 뷰
  - QEMU 명령어 표시

#### 2-4. CTest 상세 결과 표시
- **현재**: 전체 pass/fail 수만 표시
- **목표**: 각 테스트별 상태, 소요시간, 라운드(r1/r2) 표시
- **작업**:
  - `summary.build.data.ctest_results` 배열 파싱
  - 테스트별 행: 이름, 라운드, 상태, 소요시간
  - 실패 시 출력 로그 링크

#### 2-5. Coverage 파일별 상세 테이블 완성
- **현재**: 전체 rate만 표시되고 파일별 상세 미비
- **목표**: 파일별 line/branch/function 커버리지 테이블
- **작업**:
  - `coverage.xml` (Cobertura) 파싱하여 파일별 데이터 추출
  - 소스 파일별 행: 라인 rate, 브랜치 rate, 함수 rate
  - 커버리지 바 시각화
  - HTML 리포트 직접 링크

### Phase 3: 종합 보고서 자동 생성 (우선순위: 중간)

#### 3-1. 로컬 워크플로우 → DOCX/XLSX 자동 리포트
- **현재**: Docs 탭에서 수동 생성 버튼만 존재
- **목표**: 파이프라인 완료 시 자동으로 종합 문서 생성
- **작업**:
  - `analysis_summary.json` 기반 DOCX 템플릿 자동 채우기
  - 커버리지 차트, 정적 분석 요약, 퍼징 결과 포함
  - 실행 이력 비교 (이전 실행 대비 delta)

#### 3-2. 품질 게이트 판정서
- **현재**: `below_threshold` boolean만 존재
- **목표**: 상세 품질 게이트 판정 근거 문서
- **작업**:
  - 각 게이트 기준: 커버리지 ≥ 60%, 정적 이슈 error=0, 빌드 성공, 테스트 통과
  - 게이트별 PASS/FAIL + 근거 수치
  - 종합 판정: GO/NO-GO + risk_score 설명

#### 3-3. 실행 이력 비교 (Delta Report)
- **현재**: History 탭에 기본 추이 차트만
- **목표**: 이전 실행과 현재 실행의 상세 비교
- **작업**:
  - 커버리지 변화 (Δ line, Δ branch, Δ function)
  - 새로 발견된 이슈 / 해결된 이슈
  - 테스트 수 변화
  - 빌드 시간 변화

### Phase 4: 검증 (우선순위: 높음)

#### 4-1. 리포트 무결성 검증
- 모든 Phase 작업 후 파이프라인 재실행
- 각 Step에서 생성되는 파일 존재 확인
- JSON 파일 스키마 검증 (필수 필드 존재 여부)
- MD 파일 내용이 JSON 데이터와 일치하는지 확인

#### 4-2. UI 렌더링 검증
- 웹 서버 구동 후 각 탭에서 데이터 표시 확인
- 빈 데이터 / 에러 데이터에서 UI가 깨지지 않는지 확인
- DOCX/XLSX 다운로드 동작 확인

#### 4-3. 최종 통합 테스트
- 전체 파이프라인 실행 (모든 옵션 활성화)
- 생성된 모든 보고서 파일 목록 대조
- `analysis_summary.md` 내용 품질 검토
- `analysis_summary.json` 데이터 정합성 확인
