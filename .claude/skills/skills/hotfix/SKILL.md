# 긴급 버그 수정 워크플로우

$ARGUMENTS 에 버그 설명, 에러 메시지, 또는 이슈 번호가 들어옵니다.

## 자율 운영 원칙
- **전 과정 자동 실행** -- 분석->수정->테스트->커밋을 묻지 않고 완료
- 최소 변경 원칙 -- 버그 원인만 수정, 리팩토링 금지
- 3회 재시도 후 실패 시 `git stash`로 변경 보관 후 보고

## ISO 26262 안전 규칙
- 안전 관련 함수(ASIL C/D) 변경 시 reviewer 에이전트 리뷰 필수
- ASIL C/D 판별: 함수 주석의 ASIL 태그, SRS/SDS 문서의 안전 요구사항 매핑으로 확인
- 안전 관련 변경은 자동 커밋하지 않고 리뷰 결과와 함께 보고

---

### STEP 1: 버그 분석
- 에러 메시지나 증상으로 `Grep` 검색
- 원인 파일과 라인 특정
- 해당 함수의 ASIL 등급 확인
- 보고: "원인: [파일:라인] -- [설명] (ASIL: [등급/해당없음])"

### STEP 2: 수정
- 버그 원인만 정확히 수정
- 변경 범위 최소화
- ASIL C/D 함수인 경우 reviewer 에이전트에 리뷰 위임

### STEP 3: 테스트
- 관련 테스트 + 전체 단위 테스트 실행
- 비안전 테스트 실패 시 자동 수정 --> 재실행 (최대 3회)
- 안전 테스트 실패 시 수정하지 않고 보고

### STEP 4: 커밋 또는 보고
- 비안전 변경: `fix: [간결한 설명]` 형식으로 **즉시 커밋**
- 안전 변경(ASIL C/D): reviewer 결과와 함께 사용자에게 보고 (자동 커밋 안 함)
- 결과 보고:
  ```
  ## Hotfix 완료
  버그: [증상]
  원인: [파일:라인] -- [근본 원인]
  ASIL: [등급/해당없음]
  수정: [변경 내용]
  테스트: PASS
  커밋: abc1234 "fix: ..." (또는 "리뷰 대기 중")
  ```

### 롤백 전략
- 3회 재시도 후에도 테스트 실패 시:
  1. `git stash push -m "hotfix-attempt: $ARGUMENTS"` 로 변경 보관
  2. 시도한 수정 내용과 실패 원인을 상세 보고
  3. 사용자가 수동 개입할 수 있도록 `git stash list` 안내
